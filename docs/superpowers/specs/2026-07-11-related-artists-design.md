# アーティスト詳細ページ＋関連アーティスト表示 設計書

日付: 2026-07-11
ステータス: ユーザー承認済み（設計段階）

## 目的

スカウティングレポートWebアプリの名簿ビューで、アーティストを選ぶと詳細を見られ、
かつスキルの共通ワードで緩く繋がる「関連アーティスト」を発見できるようにする。
分野（カテゴリ）をまたいだ意外な組み合わせの発見も狙う。

## 決定事項（ブレスト結果）

| 論点 | 決定 |
|---|---|
| 関連度の基準 | スキル欄の共通ワード数（カテゴリは問わない、分野横断） |
| 単語一致の精度 | 部分一致（部分文字列を含む）。表記ゆれ（Aerial/Aeriel等）も緩く拾う |
| 遷移方法 | 名簿カード全体クリックで詳細ビューに切替（URL変更なし、名簿タブ内の状態遷移） |
| 表示件数 | 最大6人（共通スキル数の多い順） |

## データフローとロジック

新しい取得・ビルド処理は不要。既存の `DATA.roster.performers`（225組、ブラウザ内に
既にある）に対してクライアントサイドで計算する。

```
rosterState = { q, base, selectedName }  // selectedName を新規追加
  selectedName === null → 一覧表示（既存の検索・絞り込みビュー）
  selectedName がセット → 詳細ビュー
```

**関連度の算出（`findRelated(performer, allPerformers)`）:**
1. 対象の `skills` を `/`, `,`, `、`, `・` で分割し、trim・小文字化した単語配列にする
2. 2文字未満の単語は除外（ノイズ対策。実データ検証で `len>=3` の閾値が妥当と確認済み。
   人数タグ等の短い記号的表現を弾く）
3. 他の全アーティストについて、単語同士の**部分文字列一致**（どちらかがどちらかを含む）
   で共通語の数を数える。自分自身は除外
4. 共通語数が1以上の人を件数の多い順にソートし、上位6件を返す
5. 同数の場合は名簿の元の並び順を維持（安定ソート）
6. 0件なら空配列（詳細ビュー側で「関連アーティストは見つかりませんでした」表示）

計算コストは225人×225人の総当たり（約5万ペア、各ペアはスキル語数個の比較）で、
クリック時に同期実行しても体感遅延はない規模。メモ化は行わない（YAGNI）。

## 詳細ビュー

- 対象アーティストの情報を大きく表示: 名前・カテゴリタグ・人数タグ・スキル・メモ・
  SNSリンク（Instagram/YouTube/Contact）。情報は既存の `performerCard` と同じ項目
- 「← 一覧に戻る」ボタン: `selectedName = null` にして一覧に戻る。検索・絞り込み
  （`q`, `base`）の状態はそのまま保持される
- 「関連アーティスト」セクション: 共通スキルの多い順に最大6枚の小さめカードを表示。
  各カードをクリックすると `selectedName` がそのアーティストに切り替わり、詳細ビューが
  そのまま更新される（一覧を経由しない）
- 関連者がいない場合はセクションに「関連アーティストは見つかりませんでした」と表示

## 実装範囲

- `site/app.js` の `renderRoster`・`performerCard` を改修。`findRelated`・
  `renderDetail`（詳細ビュー描画）を新規追加
- `site/style.css` に詳細ビュー用スタイル（戻るボタン・関連カードのグリッド等）を追加
- Node側（tools/, tests/, ビルド）・データ形式（data.enc のpayload構造）の変更は一切なし
- `el()`/`replaceChildren` のみでDOM構築（innerHTML不使用の既存制約を継続）

## テスト・検証

- ブラウザで実データ（225組）を使い、以下を確認:
  - カードクリックで詳細に切替、戻るボタンで一覧に戻り検索状態が保持される
  - 関連アーティストが共通スキル数の多い順で最大6人まで表示される
  - 関連カードクリックで詳細が連鎖的に切り替わる
  - 関連者0人のケースでメッセージが出る
  - スマホ幅（375px）で崩れない
