# スカウティングレポート Webアプリ 設計書

日付: 2026-07-10
ステータス: ユーザー承認済み（設計段階）

## 目的

週次で更新しているパフォーマー・スカウティング名簿（Googleシート「Scouting report」約210名＋グループ）を、
パスワード付きのウェブアプリとしてチーム内で共有できるようにする。

## 決定事項（ブレスト結果）

| 論点 | 決定 |
|---|---|
| データの本体 | Googleシートがマスター。アプリは閲覧専用ビューア |
| 認証 | チーム共通の合言葉1つ |
| v1機能 | 名簿ビューア／週次候補リスト／更新履歴（差分）／集計ダッシュボード の4つ全部 |
| 実現方式 | GitHub Pages＋クライアントサイド暗号化（サーバー不要・無料） |

## 全体アーキテクチャ

```
Googleシート（マスター、手動編集は今まで通り）
   │  毎週月曜9:00の定期タスク（既存タスクを拡張）
   ▼
データ生成スクリプト（Node、依存最小）
   ├─ シート読取 → roster.json（名簿）
   ├─ 前回スナップショットと比較 → history.json（週次差分）
   ├─ candidates/*.md → candidates.json（新規候補）
   └─ 集計 → stats.json（カテゴリ別件数など）
   ▼
1つの data.enc に AES-GCM 暗号化（合言葉から PBKDF2 で鍵導出）
   ▼
GitHubリポジトリへ commit & push → GitHub Pages 自動公開
   ▼
チーム: URLを開く → 合言葉入力 → ブラウザ内で復号 → 閲覧
```

## セキュリティ設計

- GitHub無料プランのPagesは**公開リポジトリのみ**。よってリポジトリは公開になるが、
  名簿データは暗号化済みの `data.enc` のみを置く。**生のJSON・markdown・シートIDは
  リポジトリに含めない**（ビルドスクリプトの入力パスはローカル参照）。
- 暗号: AES-256-GCM。鍵は合言葉から PBKDF2（SHA-256、イテレーション 600,000 以上、
  ランダムソルト）で導出。ソルトとIVは data.enc に同梱（標準的な構成）。
- 合言葉は総当たり対策として長めのフレーズを設定（ユーザーが決定、コードに書かない）。
- 復号成功後、合言葉（または導出鍵）を localStorage に保存し再入力を不要にする。
  ロック（ログアウト）ボタンで消去可能。
- 合言葉変更 = 新しい合言葉で再暗号化して push するだけ。

## 画面構成（SPA・単一 index.html・スマホ対応）

1. **ロック画面** — 合言葉入力。復号成功でアプリ本体を表示。失敗時はエラー表示
2. **名簿** — カード/テーブル表示。フリーワード検索＋Base（カテゴリ）・スキル・Size（人数）で
   絞り込み。Instagram/YouTube/Contact リンク付き
3. **今週の候補** — candidates を週の新しい順に表示
4. **更新履歴** — 週ごとの差分（追加・削除・変更）をタイムライン表示
5. **ダッシュボード** — カテゴリ別件数、手薄ジャンルの可視化

技術: フレームワークなしの vanilla JS（他のアプリ app-dev/* と同じ構成）。
ビルド不要で index.html＋js/css＋data.enc のみを配信。

## リポジトリ・フォルダ構成

```
app-dev/scouting-webapp/        ← このリポジトリ（GitHubに新規作成: scouting-report-app 等）
├── docs/superpowers/specs/     ← 設計書・計画
├── site/                       ← GitHub Pages 配信対象
│   ├── index.html
│   ├── app.js / style.css
│   └── data/data.enc           ← 暗号化データ（これだけがデータの実体）
└── tools/
    └── build-data.mjs          ← ローカルデータ → JSON → 暗号化 data.enc
```

- 既存の `app-dev/scouting-report/`（archive/・candidates/・PROJECT_NOTES.md）は
  そのまま維持。build-data.mjs がそこを入力として読む。
- 合言葉はビルド時に環境変数 or ローカルの非コミットファイル（.secret 等、.gitignore）で渡す。

## 週次運用への組み込み

既存の毎週月曜9:00 定期タスク（monthly-scouting-report-expansion）の手順に以下を追記:

4. `tools/build-data.mjs` を実行して data.enc を再生成
5. commit & push（GitHub Pages に自動反映）

これにより毎週のスナップショット＋候補調査がそのままアプリの更新になる。

## エラー処理

- 復号失敗（合言葉違い・データ破損）: 「合言葉が違います」表示、再入力
- data.enc 取得失敗（オフライン等）: エラーメッセージと再読み込みボタン
- ビルドスクリプト: 入力（スナップショット/候補md）のパース失敗行は警告を出して
  スキップし、全滅時は data.enc を上書きしない

## テスト・検証

- 暗号化→復号のラウンドトリップテスト（正しい合言葉で復元、誤った合言葉で失敗）
- スナップショットmd→JSON のパーステスト（実データで件数・列の検証）
- 実データでビルドし、ブラウザプレビューで5画面すべての動作確認（検索・絞り込み・
  リンク・ロック/アンロック）をしてから GitHub Pages 公開
